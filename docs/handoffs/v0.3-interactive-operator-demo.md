# Mosaic v0.3 — Interactive Operator Demo Handoff

This is the builder handoff for the interactive, operator-driven demo increment.
It implements [`RFC-0005`](../rfcs/RFC-0005-interactive-operator-simulation.md)
and the recorded [`demo-script.md`](../demo-script.md). The completed v0.2 board
remains in [`HANDOFF.md`](../../HANDOFF.md).

## Increment goal

Turn the startup-only fixture demo into an interactive session a human operator
drives: **Start simulation** creates a synthetic session, events arrive in
configured timing, the operator triggers Analyze/briefing and records handoffs,
and a provenance/actions tab shows the full trail. Models may run live against
OpenAI, opt-in, with the deterministic fixture path as the default and fallback.
Recurrence awareness surfaces prior area handoffs and prepares reviewable notes —
never autonomous contact, never LLM self-healing.

No login and no in-app budget: the demo is open (existing public actor), and
spend is bounded by the supplied key's own provider-side limit.

## Branch and worktree model

Integration branch: `main`. Every parcel uses an isolated branch and worktree
from the latest integrated `main` SHA recorded by the coordinator:

```text
main
├─ parcel/P35-interactive-contracts
├─ parcel/P36-simulation-controller
├─ parcel/P37-domain-beat-schedule
└─ … one isolated worktree per claimed parcel
```

Do not reuse another parcel's worktree. Claim a row only when all its
prerequisites are `✅ Integrated`. Builders do not integrate their own work.

## Non-negotiable invariants (every parcel)

- Synthetic data only; no real feeds, PII, secrets, or model binaries in Git.
- The **default and fallback path is the deterministic fixture**; the demo must
  pass with no live secret present.
- Live model output **informs**; it never mutates the deterministic projection
  or issues an operational action. Invalid/refused/unavailable output is a
  recorded ModelRun that changes no state.
- Every operator action is an immutable audit record with `executed: false`,
  using the existing open public actor. No login/identity is added.
- The OpenAI key is a **server-only** runtime secret; never in the browser, Git,
  image, or committed compose file. No in-app budget governor.
- Reuse the RFC-0004 profile seam. Core packages stay free of domain identifiers.

## Per-parcel briefs

Each parcel is atomic: exclusive path ownership and an independent acceptance
check. Keep changes additive; do not alter frozen datasets, ontology schemas,
generated ontology code, migrations, or module dependencies unless the parcel
explicitly owns them. Every parcel ends with `go run ./cmd/mosaic quality`.

### P35 — interactive contracts
Add additive interfaces/values in `internal/contracts` for: an interactive
simulation session (session id, status `pending|running|ended`, ordered beats with
a configured relative schedule); a session-scoped stream event; and per-agent
model provider selection (`fixture` vs `live`). No behavior, HTTP, or storage.
Regenerate the checked-in GoMock. **Accept:** contract + mock tests compile and
prove the surface; `go vet` + `go test ./internal/contracts/...` + quality pass.

### P36 — simulation controller
Core, domain-agnostic lifecycle: start (new session, emit a clear-workspace
signal), replay beats in configured timing on a session-scoped stream, explicit
end, and reset-to-new-session that never rewrites immutable history. Timing and
clock are injected. Consumes the profile beat schedule via an interface, not a
domain import. **Accept:** tests prove start/beat-order/end/reset, that a new
session appends no duplicate immutable records, and deterministic timing with an
injected clock; `go test ./internal/simsession/...` + quality pass.

### P37 — domain beat schedule
The domesticdisturbance profile exposes its configured-timing interactive beat
sequence derived from the frozen scenario (no new dataset). **Accept:** tests
prove the beat order/timing matches the scenario and revisions 7/9 remain the
advisory pivots; package tests + quality pass.

### P38 — OpenAI transport
Implement Luna/Terra/Sol structured clients over the OpenAI Responses API behind
their existing seams. Opt-in and per-agent; server-only `OPENAI_API_KEY` read at
runtime (no downloader, no key in code/config); the deterministic fixture client
is used when live is not selected or the secret is absent. Refusals/invalid/
unavailable responses map to recorded ModelRuns with no state mutation. **Accept:**
tests use a stubbed HTTP transport (no network); prove request shaping, refusal/
timeout handling, and fixture fallback; no secret in the repo; quality passes.

### P39 — recurrence awareness
Deterministically detect prior area-specific recorded handoffs and, after a
configurable window, prepare a pending reviewable note linked to the prior
immutable records. No autonomous contact; the note is `executed:false`.
**Accept:** tests prove area matching, the configurable window, prepared-note
linkage, and that nothing is "sent"; `go test ./internal/recurrence/...` + quality.

### P40 — simulation-session API
Add start/reset/status routes and a session-scoped SSE stream over the P36
controller. Preserve existing routes, bounded serialization, and raw-payload
exclusion. **Accept:** API tests prove session start/stream/end and no-op reset
safety; `go test ./internal/api/...` + quality pass.

### P41 — operator action API
Add Analyze(Terra)/Brief(Sol)/Interpret(Luna) and approve/annotate/prepare-handoff
endpoints. Each write is an immutable `executed:false` audit record using the open
public actor. Model calls route to fixture or live per config. **Accept:** tests
prove executed:false writes, fixture-and-live routing (live via stub transport),
and that a refused/unavailable live call records a ModelRun and mutates nothing;
quality passes.

### P42–P45 — UI (serial on `ui/**`)
P42 splits the current single-file UI into an incident-first workspace with
**Start simulation** and **Analyze** and a developer status drawer; P43 adds
recipient action cards + annotation/decision controls; P44 adds the
provenance/actions tab; P45 adds the recurrence surface + live/fixture indicator.
After P42 establishes the component layout, P43/P44/P45 should own **separate
component files** so the coordinator may run them concurrently. No raw-payload or
model-response leakage; explicit recorded-vs-delivered language. **Accept per
parcel:** `npm run check`, `npm run build`, and quality pass; Svelte checks cover
the parcel's states.

### P46 — executable composition
Wire the sim controller, live/fixture model selection, and recurrence into
`cmd/mosaicdemo` startup. Server-only key handling; fixture path when no secret.
Do not change UI/API/simulator/contracts source. **Accept:** package tests prove
fixture startup unchanged and live selection is chosen only when explicitly
configured with a secret; quality passes.

### P47 — acceptance
Interactive E2E (start→beats→Analyze→handoff→provenance), a Docker runbook
update, and `docs/live-models.md` documenting opt-in, server-only key, and
fixture fallback. Exercise the image/compose without modifying them. **Accept:**
E2E + Svelte checks + full quality + a fresh isolated Docker build/start/smoke,
each reported verbatim.
